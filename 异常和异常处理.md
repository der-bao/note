#　异常和异常处理

## 一. 异常

程序运行时，代码遇到「无法正常执行的错误」，会自动抛出对象，中断代码流程。
异常的常见类别如下：

```
BaseException  # 所有异常的顶层父类
├─ SystemExit        # sys.exit() 主动退出程序
├─ KeyboardInterrupt # Ctrl+C 强制终止程序
├─ GeneratorExit     # 生成器/协程退出信号
└─ Exception         # 业务代码所有普通异常父类（重点！）
   ├─ LookupError    # 查找类异常父类
   │  ├─ IndexError # 列表/字符串索引越界
   │  └─ KeyError   # 字典key不存在
   ├─ ArithmeticError # 数学运算异常父类
   │  ├─ ZeroDivisionError # 除0
   │  └─ OverflowError    # 数值溢出
   ├─ TypeError      # 类型不匹配
   ├─ ValueError    # 类型对、值不对
   ├─ AttributeError # 无此属性/方法
   ├─ NameError     # 变量未定义
   ├─ OSError       # 系统/文件IO父类
   │  ├─ FileNotFoundError # 文件不存在
   │  └─ PermissionError   # 权限不足
   └─ ...其他常规异常
```

## 二. 异常处理

当异常发生时，可以对其进行捕获，避免打断程序的执行。Python 异常处理的完整机制由四个关键字组合而成：`try`, `except`, `else`, `finally`。

### 1. 核心语法：try-except-else-finally

```python
try:
    # 【必须包含】把你认为可能会发生报错的危险代码写在这里
    result = 10 / 0

except ZeroDivisionError as e:
    # 【可选，可写多个】精确捕获特定类型的异常。as e 表示把错误信息抓成一个变量 e
    print(f"发生除零错误: {e}")

except (TypeError, ValueError) as e:
    # 可以用元组同时捕获多种异常，只要发生其中一种就会进入此块
    print(f"发生类型或值相关的错误: {e}")

except Exception as e:
    # 【强烈建议】通常放在最后，作为“兜底”捕获其他所有未预料到的常规异常
    print(f"发生未知错误: {e}")

else:
    # 【可选】只有当 try 块里的代码【完美运行、没有任何报错】时，才会执行走这
    print(f"计算成功，结果是: {result}")

finally:
    # 【可选】无论是否出过错，最后【必定会执行】的收尾代码（哪怕 except 里写了 return 也会先执行这里）
    # 典型应用场景：用来确保清理系统资源，如：关闭打开的文件、释放网络连接、释放线程锁等。
    print("执行清理工作...")
```

**执行流程总结：**

* **如果不报错**：执行流为 `try` -> `else` -> `finally`
* **如果报了错**：执行流从报错那行中断 -> 匹配相应的 `except` -> 执行 `finally`

### 2. 主动抛出异常 (raise)

除了被动挨打（捕获系统底层的异常外），在我们自己写业务逻辑时，如果发现别人传进来的数据或系统状态极其离谱（属于非法的业务状态），我们应该主动用 `raise` 把异常丢出去，这叫做**Fail Fast 快速失败原则**。

```python
def set_age(age):
    if not isinstance(age, int):
        # 如果类型不对，直接报错，并附上详细提示
        raise TypeError("年龄必须是整数类型")  
    if age < 0 or age > 150:
        # 如果值不合法，直接报错
        raise ValueError(f"不合理的年龄值: {age}")  
  
    print(f"配置已更新年龄为: {age}")

# set_age("二十")  # 运行时主动崩溃：TypeError
# set_age(-5)     # 运行时主动崩溃：ValueError
```

### 3. 自定义异常 (Custom Exceptions)

如果你在开发一个大型项目或框架（比如你之前在改的 `hello_agents_library`），你应该定义一套属于你们项目自己特有的“业务报错名词”。
这样做的好处是，调用你代码的人一看就知道这是你业务报出的错，而不是 Python 语言本身崩了。

**核心规则**：自定义异常**必须继承自 `Exception` 类**。

```python
# 1. 定义一个顶级的业务层异常父类（通常不用写实现，就是个分类标识）
class AgentBaseError(Exception):
    """所有 Agent 相关的基础异常"""
    pass

# 2. 从上面的基类派生出具体的细分异常
class LLMConnectionError(AgentBaseError):
    """大模型连接失败报错"""
    def __init__(self, model_name, status_code):
        self.model_name = model_name
        self.status_code = status_code
        # 拼凑错误信息，然后喂给爹（Exception）
        self.message = f"连接模型 [{model_name}] 失败，状态码: {status_code}"
        super().__init__(self.message) 

# 3. 抛出与针对性捕获
try:
    raise LLMConnectionError(model_name="GPT-4", status_code=502)
except LLMConnectionError as e:
    print(f"捕获到具体的大模型错误 -> {e}")
except AgentBaseError as e:
    print("捕获到其他模糊的 Agent 业务范围错误")
```
